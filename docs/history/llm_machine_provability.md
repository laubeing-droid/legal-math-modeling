# LLM 机器证明可行性分析

> 本文档分析 juris-calculus 中每个数学证明被大语言模型机器证明的可行性。
>
> 核心区分：**计算证明**（LLM 写代码，计算机验证）vs **逻辑证明**（LLM 写推理，自己验证）

---

## 分类框架

| 证明类型 | LLM 可靠性 | 原因 |
|----------|-----------|------|
| 穷举枚举 | **极高** | LLM 写循环，计算机跑 |
| SMT/Z3 编码 | **极高** | LLM 写约束，求解器验证 |
| SymPy 符号计算 | **高** | LLM 写表达式，CAS 验证 |
| Hypothesis PBT | **高** | LLM 写属性，框架测试 |
| Lean 4 形式证明 | **中等** | LLM 草稿，Lean 验证，但当前有 sorry |
| 解析证明（手写推理） | **低** | Codex 审计发现 35% 错误率 |

**关键洞察**：凡是"计算机能跑一遍就确认"的证明，LLM 都能可靠地机器证明。凡是需要"人类读一遍推理链"的证明，LLM 不可靠。

## 核心结论

### LLM 能可靠证明的

**凡是能写成代码并运行验证的证明**，LLM 都能可靠地机器证明。这包括：
- 穷举枚举（写循环跑完所有情况）
- SMT 编码（写约束让求解器验证）
- 符号计算（写表达式让 CAS 化简）
- 性质测试（写属性让框架随机搜索）

**当前框架中约 70% 的证明属于这一类。**

### LLM 不能可靠证明的

**需要创造性数学推理的证明**，LLM 当前不可靠。Codex 审计发现 Claude 的这类证明有 35% 错误率。这包括：
- 需要"看到"关键洞察的证明
- 需要构造非显然的反例
- 需要跨领域的数学知识

**当前框架中约 30% 的证明属于这一类。**

### 未来趋势

| 时间线 | 技术 | 影响 |
|--------|------|------|
| 现在 | LLM + Z3/SymPy/Hypothesis | 计算证明已完全自动化 |
| 1-2 年 | LLM + Lean 4 + Mathlib | 形式证明大部分可自动化 |
| 3-5 年 | AlphaProof 类系统 | 创造性证明可能部分自动化 |

### 实际建议

**对你来说**，最高效的做法是：

1. **把所有证明尽量转化为计算证明**——穷举、SMT、符号计算。LLM 可以可靠地写这些代码。
2. **保留少量解析证明作为"方向指引"**——这些不需要机器证明，只需要说明"为什么这个方向是对的"。
3. **用 Lean 4 草稿标记"未来可机器证明"**——即使现在有 sorry，也比没有好。
4. **用信任标签系统记录每个证明的机器可证明性**——新增一个 `MACHINE_PROVABLE` / `HUMAN_REQUIRED` 标签。

## 对之前 7 个深挖方向的影响

| 方向 | LLM 能帮什么 |
|------|-------------|
| 非单调性论文 | LLM 可以生成反例构造代码（计算证明），但论文的核心论证需要人类写 |
| k≤3 实证检验 | **完全可自动化**——LLM 写代码解析 2,117 条规则的依赖图 |
| 真实数据 Banach | LLM 可以写 Lipschitz 常数计算代码，但数据需要人类提供 |
| 信任标签论文 | 论文本身需要人类写，但 LLM 可以生成一致性验证代码 |
| 损害赔偿属性文法 | **完全可自动化**——LLM 写属性文法求值器 |
| 博弈论 + AAF | LLM 可以写博弈求解器代码，但博弈模型设计需要人类 |
| 举证责任追踪 | LLM 可以写 Kripke 转移代码，但法律规则编码需要人类 |

**最大的自动化机会**：k≤3 实证检验和损害赔偿属性文法——两者都是"写代码跑数据"类型，LLM 可以高度自动化。
