# 03 数学模型详析：代码提升前基线

日期：2026-06-11

## 1. 总体模型

juris-calculus 的下一版数学模型应采用分层结构：

```text
Legal Data / Sources
        |
        v
Source Manifest + Trust Labels
        |
        v
Monotone Horn Closure
        |
        v
Static Argument Graph
        |
        v
Dung Grounded Extension
        |
        v
Agent Payload with Epistemic Status
```

## 2. Horn 层

允许：

1. 正向事实扩张；
2. 有界 finite KB 的 closure 验证；
3. monotone operator 的局部证明。

禁止：

1. 在同一个 Horn fixpoint 中做 rebuttal/confidence-zeroing；
2. 把 full evaluator 称为 Tarski 单调。

工程接口：

```python
horn_result = horn_closure(facts, rules)
```

输出必须包含：

1. fired rules；
2. closure set；
3. rule maturity；
4. source ids。

## 3. AAF 层

Dung AAF 用于处理：

1. rebuttal；
2. exception；
3. counter-rebuttal；
4. rebuttable presumption。

接受依据：

`P1-E1 = PROVED_BY_EXHAUSTIVE_ENUMERATION` for `n <= 4`

工程接口：

```python
aaf_result = grounded_extension(arguments, attacks)
```

输出必须包含：

1. grounded extension；
2. rejected arguments；
3. active attacks；
4. whether pattern exceeds binary AAF。

## 4. Cross-jurisdiction 层

真实数据只支持 obstruction witness，不支持全称范畴论定理。

工程规则：

1. 先查 `obstruction_registry`；
2. 若 `COLLISION` 或 `ASYMMETRY`，禁止自动映射；
3. 若 `REFERENCE_UNVERIFIED`，只可进入人工审核；
4. 若 `MATCH`，仍需保留 jurisdiction 标签。

## 5. Pricing 层

当前数据：

1. 225 条 pricing observations；
2. 全部 `fee_schedule` proxy；
3. 无 real timesheet；
4. 无 invoice。

结论：

`DATA_INSUFFICIENT_FOR_REAL_BANACH_PRICING`

工程规则：

1. pricing estimator 可实现；
2. Banach contraction 不可声称；
3. calibration 必须读取 `data_quality`；
4. real_timesheet/invoice 不足时输出降级状态。

## 6. DP 层

法律可提供：

1. release class；
2. allowed release mode；
3. approval/audit requirements。

法律不能提供：

1. 唯一 epsilon；
2. DP mechanism 的数学证明；
3. sensitivity 自动定义。

工程规则：

```python
epsilon = policy_config[data_class].epsilon_range
```

而不是：

```python
epsilon = legal_privilege_to_epsilon(privilege_level)
```

## 7. Agent 输出层

所有工具输出都必须携带：

1. `trust_label`
2. `data_origin`
3. `verification_artifacts`
4. `allowed_claim`
5. `forbidden_claim`
6. `agent_instruction`

这一步是从“数学模型”进入“工程落地”的关键。
